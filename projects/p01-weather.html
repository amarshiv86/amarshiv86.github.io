The dataset has significant class imbalance — roughly 78% No Rain, 22% Rain. Accuracy alone is misleading. The model uses class_weight="balanced" which improved recall from 41% to 75.8% compared to the unweighted baseline. ROC-AUC of 0.855 means the model correctly ranks a rainy day above a dry day 85.5% of the time.

Architecture

Training and deployment are fully decoupled — pushing a new model does not redeploy the API, and updating the API does not retrain the model. The Space pulls model.pkl from HF Hub at startup, so it always serves the latest model without rebuilding the container.

How it works

Given 20 weather measurements — temperature, humidity, pressure, cloud cover, wind direction and speed — the model predicts whether it will rain the following day. The full pipeline runs automatically when source code or hyperparameters change.

  • 1
    prepare.py — reads 145k rows from Bureau of Meteorology CSV, handles missing values, encodes wind directions (N/S/E/W → integers), splits train/test
  • 2
    train.py — builds sklearn Pipeline (StandardScaler → RandomForest), trains with params from params.yaml, logs run to MLflow
  • 3
    evaluate.py — computes accuracy, F1, ROC-AUC, generates confusion matrix, writes metrics.json
  • 4
    hf_upload.py — pushes model.pkl + metrics to HuggingFace Hub model registry for versioned storage
  • 5
    FastAPI app — downloads model from HF Hub on startup, serves HTML form UI + /predict/json endpoint + /health

Key Design Decisions
Why RandomForest over XGBoost or a neural network?
RandomForest handles mixed numeric/categorical features naturally, is robust to outliers, and gives calibrated probability outputs needed for the confidence score. For tabular weather data with 20 features it outperforms neural networks which need far more data to generalize.
Why sklearn Pipeline instead of separate scaler + model?
Wrapping StandardScaler and RandomForest in a single Pipeline means the scaler is fitted only on training data and automatically applied at inference — preventing data leakage. The entire pipeline serializes into one model.pkl file.
Why HF Hub as model registry instead of MLflow model registry?
MLflow model registry requires a running server. HF Hub is free, versioned, and publicly accessible with a Python SDK. The FastAPI app downloads the model at startup with one line — no server infrastructure needed.
Why FastAPI over Gradio for serving?
Gradio wraps everything in its own UI with limited customization. FastAPI gives full control over the HTML, endpoints, and response format. The /predict/json REST endpoint and custom HTML form UI are only possible with FastAPI.

Try it yourself
💤 HF Space is paused to stay within free tier limits. Visit the Space and click Restart this Space (takes ~30 seconds), then use the curl commands below.

Once the Space is running, call the JSON API directly: 

# JSON prediction endpoint
curl -X POST https://amarshiv86-weather-aus-rain-api.hf.space/predict/json \
  -H "Content-Type: application/json" \
  -d '{"MinTemp":13.4,"MaxTemp":22.9,"Rainfall":0.6,
     "Evaporation":5.0,"Sunshine":7.5,"WindGustDir":"S",
     "WindGustSpeed":44,"WindDir9am":"W","WindDir3pm":"SW",
     "WindSpeed9am":20,"WindSpeed3pm":24,"Humidity9am":71,
     "Humidity3pm":22,"Pressure9am":1007.7,"Pressure3pm":1007.1,
     "Cloud9am":4,"Cloud3pm":5,"Temp9am":16.9,"Temp3pm":21.8,
     "RainToday":0}'

# Response
{ "prediction": "No", "confidence_pct": 62.5, "latency_ms": 12 }

# Health check
curl https://amarshiv86-weather-aus-rain-api.hf.space/health
