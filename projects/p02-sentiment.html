ROC-AUC of 0.960 means the model correctly distinguishes positive from negative sentiment 96% of the time. The model is significantly stronger than P01's classifier because transformer attention mechanisms capture long-range dependencies in text that simple feature engineering cannot.

How it works

Four stages run automatically on every push — training, evaluation, drift detection, and deployment are all decoupled:

  • 1
    prepare.py — downloads IMDB from HF Hub, strips HTML tags from reviews, subsamples for fast iteration (configurable via params.yaml), saves to train/test CSV
  • 2
    train.py — fine-tunes distilbert-base-uncased using HuggingFace Trainer API, logs all params and metrics to MLflow, saves best checkpoint by F1 score
  • 3
    evaluate.py — runs full evaluation on held-out test set, generates confusion matrix PNG, writes metrics.json with accuracy/F1/ROC-AUC/precision/recall
  • 4
    drift_report.py — runs model inference on both train and test samples, compares distributions across 6 columns using Evidently AI, saves HTML drift report
  • 5
    hf_upload.py — pushes model weights + tokenizer to HF Hub model registry, pushes processed dataset to HF Hub dataset registry
  • 6
    FastAPI app — downloads model from HF Hub at startup, serves HTML form + /predict/json + /health endpoints
Architecture

P02 adds a fourth pipeline stage that P01 doesn't have — drift detection. After training and evaluation, drift_report.py runs inference on both training and test samples and uses Evidently AI to compare their distributions. The report is saved as an HTML artifact and pushed to HF Hub alongside the model.

Drift Detection — How it works

Evidently AI compares the reference data (training set sample) against current data (test set, simulating production traffic) across 6 columns. Each column gets a statistical test appropriate for its type:

label distribution
chi-squared test
✓ No drift
model predictions
chi-squared test
✓ No drift
confidence scores
Kolmogorov-Smirnov
⚠ Slight shift
text length
Kolmogorov-Smirnov
✓ No drift
word count
Kolmogorov-Smirnov
✓ No drift
Dataset drift detected
16.7% columns drifted
✓ False — model stable

In production you'd run this comparison between your training data and a rolling window of incoming requests. If drift is detected, the pipeline automatically triggers retraining. This is the standard MLOps pattern for catching concept drift before it degrades model accuracy.

Key Design Decisions
Why DistilBERT over full BERT or a simpler model?
DistilBERT is 40% smaller and 60% faster than BERT while retaining 97% of its performance. For sentiment classification on short-to-medium text it's the best trade-off between accuracy and inference cost. Full BERT would be slower without meaningful accuracy gain on IMDB reviews.
Why Evidently AI for drift detection over custom code?
Evidently AI automatically selects the appropriate statistical test per column type (chi-squared for categorical, KS for continuous), generates a visual HTML report, and has a clean Python API. Writing equivalent drift detection from scratch would take far longer and produce worse visualizations.
Why compare training vs test for drift instead of real production data?
This project simulates the production drift monitoring pattern. In a real deployment you'd compare training data against a rolling window of incoming requests. The test set here acts as a proxy for production traffic to demonstrate the pipeline without needing live traffic.
Why add text_length and word_count to drift monitoring?
Input length distribution is an early signal of data source changes. If production reviews suddenly get much shorter (e.g. from a different platform), the model's confidence distribution will shift even before accuracy degrades. These features act as leading indicators.
