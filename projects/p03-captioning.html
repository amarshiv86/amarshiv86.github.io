Model Architecture

P03 uses a pre-trained VisionEncoderDecoderModel — a HuggingFace architecture that bridges vision and language models. The encoder and decoder are two separate specialist models connected by a cross-attention mechanism:

Encoder · Vision
Vision Transformer (ViT)
Splits the image into 16×16 pixel patches, treats each patch as a token, and runs self-attention across all patches to build rich spatial representations. Outputs a sequence of patch embeddings capturing both local and global image features.
Decoder · Language
GPT-2
Takes the ViT patch embeddings as context via cross-attention and auto-regressively generates caption tokens one at a time. Beam search explores multiple candidate sequences simultaneously and selects the highest-probability caption.

The combined model is nlpconnect/vit-gpt2-image-captioning, pre-trained on the COCO dataset — 118,000 images each with 5 human-written captions. No custom training was done in this project — the focus is on the inference pipeline, UI, and deployment.

Architecture
How it works
  • 1
    Image upload — user uploads any image via the Gradio UI. The image is loaded with PIL and converted to RGB.
  • 2
    ViT encoding — the image is split into 16×16 patches, each converted to a 768-dimensional embedding. The ViT processes all patches with self-attention, producing a 196-token sequence of patch representations.
  • 3
    Cross-attention — the GPT-2 decoder attends to the ViT patch embeddings to condition its text generation on the image content. Each generated token can attend to every image patch.
  • 4
    Beam search — instead of greedily picking the most likely next token, beam search maintains N candidate sequences simultaneously. Higher beam width = more exploration = better captions at the cost of compute.
  • 5
    Caption output — the decoder generates up to 16 tokens, producing a natural language sentence describing the image. Multiple captions can be generated to see model uncertainty.

The model generates descriptive captions covering objects, actions, spatial relationships, and scene context — learned from 118k COCO image-caption pairs:

Increasing beam search width improves caption quality — beam width 4 typically produces more grammatically complete sentences than width 1 (greedy decoding). Generating multiple captions lets you see the top-N hypotheses the model considered.

Features
Beam search control — width 1–8 via slider, higher = more accurate
Multiple captions — generate up to 4 candidates to compare confidence
GPU auto-detect — uses CUDA if available, falls back to CPU
Any image format — PNG, JPG, WEBP all supported via PIL
Pre-trained model — no training required, downloads from HF Hub
Auto-deploy — GitHub Actions deploys to HF Spaces on every push
Key Design Decisions
Why use a pre-trained model instead of training from scratch?
Training an image captioning model from scratch requires large GPU compute (days on A100s) and a massive dataset. The pre-trained nlpconnect/vit-gpt2 model was trained on COCO's 118k images. For a portfolio project demonstrating multimodal inference, using a strong pre-trained model and building the deployment pipeline around it is the right engineering trade-off.
Why ViT + GPT-2 over BLIP or other captioning models?
ViT+GPT-2 cleanly demonstrates the encoder-decoder architecture — two separate specialist models connected by cross-attention. BLIP uses a more complex bootstrapped training approach that's harder to explain. For a portfolio project, architectural clarity matters as much as benchmark performance.
Why expose beam search width as a user control?
Beam search width is a core inference hyperparameter. Exposing it lets users experience the quality-speed trade-off directly — width 1 is instant but often produces clipped sentences, width 8 takes 2x longer but produces complete descriptive captions. This makes the demo educational, not just functional.
