RAG Pipeline

Why RAG?
❌ Without RAG
LLM answers from training data only. For document-specific questions it either hallucinates plausible-sounding wrong answers or says "I don't know." It has no access to your specific document.
✓ With RAG
The LLM is given the actual relevant passages from your document as context. It can only answer based on what's in the document — grounded, accurate, and document-specific.

RAG solves the fundamental limitation of LLMs — they can't know what's in your document. By retrieving relevant chunks first and passing them as context, the LLM becomes a reading comprehension engine rather than a knowledge recall engine.

How it works
  • 1
    PDF extraction — pypdf reads each page and concatenates text. HTML/formatting artifacts are stripped. Empty PDFs (scanned images without OCR) are caught and reported.
  • 2
    Text chunking — LangChain's RecursiveCharacterTextSplitter splits text into 500-token chunks with 50-token overlap. The overlap ensures no sentence is split across a chunk boundary, preserving context.
  • 3
    Embedding — all-MiniLM-L6-v2 converts each chunk into a 384-dimensional dense vector. This model runs locally (no API cost), is fast on CPU, and is optimized for semantic similarity search.
  • 4
    ChromaDB indexing — chunk vectors are stored in a local ChromaDB instance at /tmp/chroma_db. ChromaDB uses approximate nearest neighbor search for fast retrieval.
  • 5
    Question retrieval — the user's question is embedded with the same model. ChromaDB returns the top-3 most semantically similar chunks using cosine similarity.
  • 6
    Grounded generation — the 3 retrieved chunks are injected into a prompt sent to Llama-3.2-1B-Instruct via HF Inference API. The prompt instructs the model to answer only from the provided context.
Architecture

The pipeline has two phases — indexing (done once on upload) and querying (done on every question). The same embedding model is used in both phases to ensure the question and document vectors are in the same semantic space. ChromaDB is shared between both paths — indexed chunks are retrieved during query time.

Key Design Decisions
Why all-MiniLM-L6-v2 for embeddings instead of OpenAI embeddings?
all-MiniLM-L6-v2 runs locally with no API cost — critical for a demo that processes arbitrary user PDFs. It's a 22M parameter model optimized for semantic similarity tasks, producing 384-dim vectors. For document Q&A retrieval it performs comparably to much larger embedding models.
Why chunk_size=500 with chunk_overlap=50?
500 tokens fits comfortably within the LLM's context window while capturing enough context to answer most questions. The 50-token overlap prevents answers from being split across chunk boundaries — without overlap, a sentence starting near a chunk end and finishing at the next chunk start would never be retrieved as a complete unit.
Why Llama-3.2-1B via HF Inference API instead of a local model?
Running a 1B+ parameter LLM locally on CPU takes 30-60 seconds per query — too slow for a demo. The HF Inference API serves the model on GPU infrastructure for free (with HF_TOKEN), giving sub-5-second responses. This is the correct production pattern: local embeddings + cloud LLM.
Why top-3 chunks instead of more?
Retrieving more chunks risks including irrelevant content that confuses the LLM, a phenomenon called "lost in the middle" — LLMs tend to ignore context in the middle of long prompts. Top-3 keeps the context focused and within a reasonable prompt length for the 1B model's context window.
