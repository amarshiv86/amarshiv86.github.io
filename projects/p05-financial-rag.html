What makes P05 different from P04

P04 was a user-driven RAG — upload your own PDF, ask questions. P05 is an automated pipeline RAG — the system continuously ingests fresh data on a schedule, maintains a persistent vector store, and serves it via a proper REST API with multiple endpoints. This is much closer to how production RAG systems actually work.

ℹ️ Honest implementation note: The "answer" returned by the API is a structured summary of retrieved article titles and snippets — not LLM generation. The embedding uses a custom lightweight MD5 hash-based vector (128-dim) rather than a neural embedding model, making the ingest pipeline run in CI without GPU or heavy dependencies. This is an intentional engineering trade-off for reliability over accuracy.

Architecture
How it works
  • 1
    GitHub Actions cron — triggers ingest.py daily at 6am UTC. Also runs on every push to src/ for testing.
  • 2
    RSS fetch — feedparser pulls up to 20 entries from each of 5 feeds. Failed feeds are caught and skipped without stopping the pipeline.
  • 3
    Deduplication — MD5 hash of article URL checked against ingested_ids.json. Only new articles proceed to embedding. Prevents the index from growing with duplicates on each daily run.
  • 4
    Lightweight embedding — custom 128-dim word frequency vector using MD5 hashing. No torch, no GPU, no external API — runs in GitHub Actions CI without heavy dependencies.
  • 5
    ChromaDB indexing — articles stored with metadata: source, link, date, detected tickers. Persistent ChromaDB at data/chroma_db/.
  • 6
    Push to HF Dataset — updated ChromaDB pushed to amarshiv86/financial-news-db HF Dataset. This is the "database" that the API reads from.
  • 7
    FastAPI startup — API downloads ChromaDB from HF Dataset at startup. Subsequent restarts use the cached local copy. Serves /query, /news, /health endpoints.

Key Design Decisions
Why a custom MD5 hash embedding instead of all-MiniLM?
The ingest pipeline runs in GitHub Actions CI — no GPU, limited memory. Installing torch + sentence-transformers would make the CI job slow and fragile. The MD5 word-hash embedding (128-dim) requires zero external dependencies, runs in milliseconds, and is good enough for keyword-based financial news retrieval where exact ticker names and company names matter more than semantic similarity.
Why store ChromaDB on HF Dataset instead of in the Space?
HF Spaces have ephemeral storage — every restart wipes /tmp. By pushing ChromaDB to an HF Dataset, the indexed articles survive Space restarts. The API downloads the DB at startup and caches it locally. This is the correct pattern for stateful data in stateless container environments — same principle as S3-backed databases.
Why no LLM for answer generation?
Financial news retrieval is a precision task — users want the actual headline and summary, not a paraphrased version. An LLM would add latency and risk hallucinating financial figures. Structured retrieval (article title + summary + source link) is more trustworthy for financial information than generated prose.
Why ticker filtering at query time rather than index time?
Filtering at query time is more flexible — the same index serves both filtered (NVDA news only) and unfiltered (all market news) queries. The ticker tags in metadata make filtering cheap: after similarity search, results are filtered by ticker string match in metadata, requiring no re-embedding.

Try it yourself
💤 HF Space is paused to stay within free tier limits. Visit the Space and click Restart this Space (takes ~30 seconds). 
# Run locally
git clone https://github.com/amarshiv86/p05-financial-news-rag
cd p05-financial-news-rag
pip install -r requirements.txt

# 1. Ingest latest news
python src/ingest.py
# → Fetches from 5 RSS feeds, deduplicates, indexes into ChromaDB

# 2. Start API
export CHROMA_DIR=data/chroma_db
uvicorn api.app.main:app --reload --port 7860

# 3. Query the API
curl "http://localhost:7860/query?q=latest+NVDA+news&ticker=NVDA"
curl "http://localhost:7860/news?ticker=AAPL&limit=5"
curl "http://localhost:7860/health"
