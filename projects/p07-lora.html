ROUGE measures overlap between generated and reference answers. The dramatic improvement in ROUGE-2 (+153.6%) means the fine-tuned model produces bigrams (two-word phrases) that match the reference answers — technical terms like "kubectl logs", "burn rate", "error budget" appear in the correct context. The base model produces generic responses that miss domain-specific terminology entirely.

Q: What steps should I take when a pod is in CrashLoopBackOff?
❌ Base Phi-3-mini
A CrashLoopBackOff error means the container is crashing repeatedly. You should check what is wrong with your application.
✅ Fine-tuned
1. Check pod logs: kubectl logs <pod> --previous to see the crash reason. 2. Describe the pod: kubectl describe pod <pod> to review events and exit codes. 3. Exit code 137 means OOMKilled — increase memory limits. 4. Verify the container image is pullable. 5. Review liveness probe settings — misconfigured probes trigger restart loops.
Q: How do I calculate error budget remaining for a 99.9% SLO?
❌ Base Phi-3-mini
Error budget is the amount of downtime you are allowed.
✅ Fine-tuned
Error budget = 1 - 0.999 = 0.1% allowed failure. For a 30-day window: 0.001 x 30 x 24 x 60 = 43.2 minutes of allowed downtime. Monitor burn rate — if burn rate exceeds 1 you are consuming budget faster than it replenishes.
How QLoRA Fine-tuning Works
  • 1
    4-bit quantization — Phi-3-mini loaded in NF4 (normalized float 4-bit) using BitsAndBytes. Reduces the 3.8B parameter model from ~7GB to ~2GB VRAM, making it trainable on a free Colab T4 GPU.
  • 2
    LoRA adapter injection — PEFT adds small trainable matrices (rank 16) to the q_proj and v_proj attention layers. Only ~1% of parameters are trained — the base model weights stay frozen.
  • 3
    Phi-3 instruction format — training examples formatted as <|user|> question <|end|> <|assistant|> answer <|end|> to match the model's pre-training format. Mismatched format reduces fine-tuning effectiveness.
  • 4
    Training — 3 epochs, batch size 2, gradient accumulation 4 (effective batch 8), learning rate 2e-4 with cosine scheduler and 3% warmup. BF16 mixed precision on supported hardware.
  • 5
    ROUGE evaluation — before/after comparison on held-out questions. ROUGE-1/2/L scores computed against reference answers to quantify improvement.
  • 6
    Adapter push to Hub — only the LoRA adapter weights (~50MB) pushed to HF Hub, not the full 3.8B model. At inference time, adapter is merged with the base model.

Key Design Decisions
Why Phi-3-mini instead of Llama 3 or Mistral?
Phi-3-mini (3.8B) fits in 2GB VRAM with 4-bit quantization — the free Colab T4 GPU has 15GB but this leaves headroom for activations and gradients. Llama 3 8B would require 8GB+ even quantized, risking OOM errors on free tier. Phi-3-mini also has strong instruction following from Microsoft's training, making it more responsive to fine-tuning on small datasets.
Why LoRA rank 16 instead of higher or lower?
Rank 16 gives a good balance for a 15-example dataset. Lower rank (4-8) may underfit — not enough capacity to learn the SRE domain patterns. Higher rank (32-64) risks overfitting on 15 examples and also increases memory usage. The lora_alpha=32 (2x the rank) is the standard scaling factor recommendation from the LoRA paper.
Why only q_proj and v_proj as target modules?
The query and value projections in attention control what the model focuses on and how it combines information. Fine-tuning these two is sufficient to steer the model toward domain-specific outputs while keeping the parameter count minimal. Adding k_proj or o_proj increases parameters without proportional improvement on small datasets.
Why 15 training examples — isn't that too few?
Yes and no. 15 examples is minimal, but the goal is demonstrating the fine-tuning pipeline and that domain adaptation is possible even at small scale. The 65.8% ROUGE-1 improvement proves the technique works. In production you'd use 500-5000 examples for robust performance. The small dataset also makes training accessible — anyone can replicate this on free Colab.
