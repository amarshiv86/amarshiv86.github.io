The ReAct Pattern

ReAct (Reason + Act) is the standard pattern for LLM agents that use tools. Instead of answering directly from training data, the agent loops through three phases until it has enough information to answer:

Phase 1
🤔 Reason
→
Phase 2
🔧 Act
→
Phase 3
👁 Observe
→
Repeat
🤔 Reason

The LLM decides which tool to call and with what arguments based on the query and previous observations. After each tool call it sees the result and decides whether to call another tool or produce a final answer. This continues for up to 5 steps.

5 Tools

Each tool is a Python function registered in the TOOLS dict with a description, argument spec, and requires_approval flag:

check_slo_status
Checks SLO burn rate and error budget for a service. Returns status (OK/ELEVATED/WARNING/CRITICAL), burn rate 1h/6h, budget remaining %. Mock Prometheus data for 4 services.
No approval
get_runbook
Fetches runbook steps for an alert name. Real file lookup from runbooks.jsonl — supports exact and fuzzy matching. Returns alert name, severity, steps, escalation path.
No approval
query_recent_alerts
Returns currently firing alerts filtered by severity. Mock PagerDuty data — 3 realistic alerts with service, duration, summary. Filterable by critical/warning/all.
No approval
calculate_error_budget
Pure math — calculates allowed downtime for any SLO target and window. Returns budget in minutes and hours, consumed vs remaining. Always works, no dependencies.
No approval
check_service_health
Real HTTP call to any URL — returns status code, latency, response size. The only tool that makes external network calls, which is why it requires human approval before executing.
⚠ Requires approval
Human-in-the-Loop Approval

When the agent decides to call check_service_health, it pauses and presents the pending call to the user for approval. The agent state is preserved — approve continues from exactly where it stopped, reject sends the error back as a tool result: 

🤔 Thought: I need to check the auth-service error rate first.

🔧 Tool: check_slo_status → {"service_name": "auth-service"}

✅ Result (2ms): {
  "service": "auth-service",
  "status": "CRITICAL",
  "burn_rate_1h": 4.5,
  "budget_remaining_pct": 97.2,
  "action_required": "Investigate — exhausts budget in <5 days"
}

🤔 Thought: CRITICAL status. Let me check the health endpoint directly.

🔧 Tool: check_service_health → {"url": "https://auth-service.internal/health"}

⚠️ Approval needed for: check_service_health({"url": "https://auth-service.internal/health"})

✅ Approved: check_service_health(...)

✅ Result (312ms): {
  "status_code": 200,
  "healthy": true,
  "latency_ms": 312
}

ANSWER: auth-service is CRITICAL — burn rate 4.5x (SLO: 99.9%).
Health endpoint responds 200 in 312ms. Error rate 0.45% vs 0.1% threshold.
Action: check recent deployments and review slow query logs.

Latency tracking — every ToolResult includes latency_ms for observability
Max steps guard — hard limit of 5 steps prevents infinite loops
Graceful tool errors — unknown tools and bad args return ToolResult errors, not crashes
Real runbook lookup — get_runbook reads actual runbooks.jsonl file, not hardcoded
Burn rate logic — CRITICAL/WARNING/ELEVATED/OK mapped from real SRE thresholds
State preservation — agent state survives approval pause, resumes from exact step
Key Design Decisions
Why human-in-the-loop for only check_service_health?
The four read-only tools (SLO status, runbook lookup, alerts, error budget math) carry zero risk — they either read local files or compute math. check_service_health makes real external HTTP calls: it could hit internal endpoints, trigger rate limits, or expose which URLs the agent is checking. Requiring approval for external calls follows the principle of least privilege — the agent only gets network access when a human explicitly grants it.
Why Qwen2.5-0.5B instead of a larger model or API?
0.5B parameters fits in CPU memory on the HF Space free tier (16GB RAM). An API like Mistral or GPT-4 would give better tool-use accuracy but introduces latency, cost, and an external dependency. For a portfolio demo the goal is showing the agent architecture — the ReAct loop, tool registry, approval system, and state management work regardless of model quality. Larger models are a drop-in replacement.
Why parse structured TOOL/ARGS format instead of function calling?
Qwen2.5-0.5B doesn't reliably support JSON function calling — it's too small. The TOOL/ARGS text format with regex parsing is more robust for small models. The system prompt shows the exact format with an example, and the parser strips markdown fences and finds the JSON block. This approach works on any model, including ones that don't support the OpenAI function calling spec.
Why a ToolResult dataclass instead of raw dicts?
ToolResult enforces a consistent interface — every tool returns the same fields (tool, success, data, error, latency_ms) regardless of what it does internally. This makes the agent loop simple: it never needs to know which tool it called to display the result correctly. The trace renderer treats all ToolResult objects identically, and testing is straightforward — mock any tool by returning a ToolResult.
