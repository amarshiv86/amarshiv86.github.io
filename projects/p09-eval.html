LLM Eval Framework

Generic evaluation framework for LLM responses using three complementary methods — ROUGE, LLM-as-judge, and a custom domain rubric. SQLite tracks every run for regression detection. GitHub Actions CI gate blocks deploy if the composite score drops more than 5%.

Three Complementary Eval Methods

No single method is sufficient — ROUGE misses semantic quality, LLM judges have bias, rubrics miss edge cases. Combining all three gives a more robust composite score:

40%
LLM-as-Judge
Local Qwen2.5-0.5B scores each response 0–10 with reasoning. Falls back to heuristic (ROUGE-based) in CI where no model is loaded — no blocked pipelines.
35%
Custom Rubric
Domain-specific heuristic scoring across dimensions like accuracy, actionability, completeness. Three rubric variants: general, SRE, and RAG. No model required.
25%
ROUGE
Text overlap baseline using ROUGE-1, ROUGE-2, and ROUGE-L. Fast, deterministic, always consistent. Acts as a sanity check on the other two methods.

Composite score = ROUGE (25%) + LLM Judge (40%) + Rubric (35%). A response passes if the composite reaches 6.0 / 10.

Custom SRE Rubric

The rubric has three domain-specific variants. The SRE category explicitly checks for kubectl commands, escalation paths, and actionable steps — things generic eval frameworks miss entirely:

The SRE rubric's actionability dimension checks for kubectl, grep, systemctl, docker, helm, and prometheus commands. The escalation dimension checks for words like "escalate", "page", "on-call", "severity". Generic eval frameworks score these the same as any other text — the SRE rubric rewards domain-appropriate responses.

Evaluation Pipeline
  • 1
    Load test cases — reads test_cases.jsonl. Each case has: id, question, reference_answer, generated_answer, category (general/SRE/RAG).
  • 2
    ROUGE scoring — rouge_scorer computes ROUGE-1/2/L F1 scores against the reference answer. Fast, no model needed.
  • 3
    LLM judge — Qwen2.5-0.5B prompted to score 0–10 with a one-line reason. Falls back to ROUGE-based heuristic if no model loaded (CI mode).
  • 4
    Rubric scoring — heuristic scoring across 4 domain dimensions based on word overlap, length ratio, command detection, and escalation keywords.
  • 5
    Composite score — weighted average: ROUGE 25% + LLM judge 40% + rubric 35%. Pass if ≥ 6.0/10.
  • 6
    Save to SQLite — every run saved to eval_history.db with all per-test scores and run summary. Used by P10 Observability Dashboard.
  • 7
    Regression check — compares current avg_composite against previous run. If drop > 5%, CI gate fails and deployment is blocked.
  • 8
    Markdown report — generates eval_report.md with per-test table, regression status, and score weights.
Architecture
CI Gate — How it works

The CI gate runs as the last step of the GitHub Actions eval workflow. It reads the latest two runs from SQLite, computes the score drop, and exits with code 1 if regression is detected — blocking deployment: 

── CI Gate ─────────────────────────────
Run ID:    run_20260530_142301_a3f2c1
Score:     5.69 / 10
Pass rate: 40.0%
Status:    Score OK — dropped 2.1% (within 5% threshold)
────────────────────────────────────────

✅ CI GATE PASSED

# If regression detected:
❌ CI GATE FAILED — score regression detected
   Deployment blocked. Fix the model or update test cases.

The threshold is configurable via --regression-threshold (default 5%). The gate compares avg_composite — the weighted average across all test cases — not individual test results. A single test failing won't block deploy unless it drags the composite below the threshold.

Key Design Decisions
Why SQLite instead of a proper database?
SQLite requires zero infrastructure — no server, no connection string, no credentials. The eval_history.db file can be committed to git, copied between machines, and read by the P10 observability dashboard without any setup. For eval score tracking (low write volume, small data) SQLite is the correct tool. "Zero infra — single file, works everywhere" is explicitly in the code comments.
Why 40% weight for LLM judge over ROUGE?
ROUGE measures token overlap — a semantically correct answer that uses different words gets a low ROUGE score. For SRE answers, "kubectl logs pod-name --previous" and "check previous container logs with kubectl" are equally correct but ROUGE scores them very differently. The LLM judge captures semantic correctness that ROUGE misses, so it gets the highest weight.
Why a heuristic rubric instead of another LLM judge?
Two LLM judges would have correlated errors — both might miss the same failure modes. A rule-based rubric is fully independent from the LLM judge: it can't be influenced by model phrasing biases and always runs deterministically with zero latency. The SRE-specific keyword checks (kubectl, escalation terms) catch domain-specific failures that no general-purpose LLM judge would be calibrated for.

Try it yourself

The HF Space is always running — paste any question, reference answer, and generated answer to see the full eval breakdown: 

# Run locally
git clone https://github.com/amarshiv86/p09-llm-eval
cd p09-llm-eval
pip install -r requirements.txt

# Run eval on test cases
python -m src.run_eval --dataset data/raw/test_cases.jsonl

# CI gate check (reads latest DB run)
python -m src.reporter --ci

# Run with custom regression threshold
python -m src.run_eval --regression-threshold 10.0

# Run tests
pytest tests/ -v
