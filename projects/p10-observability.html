Latency p50/p95/p99

Daily cost breakdown by model — qwen2.5-0.5b, phi-3-mini, mistral-7b. 7-day trend with cost spike detection (alerts if today is 50%+ above yesterday).

6 prompt versions tracked across p06, p08, p09 — each with avg score, request count, and description. Shows how prompt changes affect quality over time.
Anomaly Alerts
4 alert detectors run on each refresh: SLO burn, latency breach, cost spike, eval regression. Sorted critical → warning → info with actionable recommendations.
Data Sources

REAL
P09 eval_history.db
SQLite file from P09 copied to data/processed/. Dashboard reads actual eval runs, scores, and pass rates via direct SQL query.
MOCK

MOCK

Realistic latency distributions for 3 AI services with ±15% random jitter on each refresh to simulate live data.
MOCK

Anomaly Detection

Four independent detectors run on every dashboard refresh. Each returns structured Alert objects with service, message, value, threshold, and an actionable recommendation:

DetectorTrigger conditionSeverityRecommendation
SLO burn rate burn_rate > 14.4x CRITICAL Page on-call immediately. Check recent deploys.
SLO burn rate burn_rate > 6x WARNING Investigate error rate. Review slow query logs.
Latency p99 > 100% of SLO CRITICAL Check for slow queries or resource saturation.
Latency p99 > 90% of SLO WARNING Monitor closely — approaching SLO breach.
Cost spike Today > yesterday × 1.5 WARNING Check for unusual traffic or prompt loops.
Eval regression Score drop > 10% CRITICAL Block deployment. Review prompt changes.
Eval regression Score drop > 5% WARNING Investigate prompt changes. Run extended suite.

Key Design Decisions
Why use real P09 data instead of all-mock?
A dashboard with 100% synthetic data doesn't prove anything — it's a UI demo. Using the actual P09 eval_history.db means the eval scores panel shows real run history, real score trends, and real pass rates. When a recruiter or interviewer looks at the dashboard they're seeing genuine data from a real eval run, not hardcoded numbers that always look good.
Why a single get_dashboard_data() call instead of per-panel fetches?
Per-panel fetches would hit the SQLite DB and run anomaly detection 6 separate times per refresh — wasteful and potentially inconsistent (panel 1 and panel 6 would show data from different moments). A single collector call snapshots all data atomically, then each panel renders from the same snapshot. This is how production dashboards work — one scrape, multiple views.
Why track prompt versions in the observability dashboard?
Prompt changes are deployments — they change model behavior as much as code changes do. Tracking prompt version history alongside eval scores lets you correlate "eval score dropped on this date" with "prompt v1.0 was deployed on this date". Without prompt versioning, debugging quality regressions in AI systems is guesswork. This is a gap in most AI monitoring tools.
