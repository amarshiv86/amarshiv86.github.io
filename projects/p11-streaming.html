Why TTFT Matters

Time To First Token (TTFT) is the most important latency metric for streaming LLM APIs. It measures the time from request sent to first token received — the moment the user sees the response start appearing. This is what makes a streaming API feel fast or slow, regardless of total generation time.

In production you'd SLO on p95 TTFT < 500ms for GPU-served models. On CPU (this demo) TTFT is ~33 seconds — expected and clearly displayed. The metrics infrastructure is production-ready regardless of the hardware.

Four Core Features
TTFT + Throughput Tracking
Every request gets a StreamMetrics dataclass tracking: started_at, first_token_at, completed_at, token_count. TTFT, total duration, and tokens/second computed as properties. Last 50 requests stored in MetricsStore for p95 calculation.
Cancellation Tokens
Each session gets a CancellationToken — user clicks Stop, token is set to cancelled, next iteration of the streaming loop checks it and exits cleanly. State recorded: tokens generated before cancel, latency up to cancellation point.
Per-user Rate Limiting
Token bucket rate limiter — max 10 requests per 60 seconds per session ID. Thread-safe with a lock. Expired entries pruned on each check. Returns retry_after seconds when limit hit — not just a 429.
Real-time Metrics Panel
Dashboard shows live stats from MetricsStore: total requests, completed, cancelled, errors, avg TTFT, p95 TTFT, avg throughput. Updates on every request — no polling needed.
Architecture
How it works
  • 1
    Rate check — RateLimiter.is_allowed(session_id) called first. If limit hit, yields a rate limit message with retry_after seconds. No request proceeds.
  • 2
    StreamMetrics created — new dataclass with request_id (UUID), started_at timestamp. CancellationToken created for this session.
  • 3
    Model generates — Qwen2.5-0.5B runs full inference (transformers doesn't support true token streaming). Full output stored in memory.
  • 4
    Word-by-word streaming — output split by spaces, yielded one word at a time with 20ms delay. First word triggers record_first_token() — TTFT measured here.
  • 5
    Cancellation check — every iteration checks cancel_token.is_cancelled. If set, calls record_cancel() and exits the generator cleanly.
  • 6
    Metrics update — on complete or cancel, StreamMetrics saved to MetricsStore. Dashboard panel reads from MetricsStore.summary() on next render.

SRE Additions
TTFT tracking — first_token_at timestamp captured on first yielded token
p95 TTFT — sorted percentile computed from MetricsStore history
Cancellation tokens — clean exit preserving partial metrics
Thread-safe rate limiter — threading.Lock prevents race conditions
retry_after — rate limit returns exact seconds until window resets
StreamMetrics dataclass — typed, consistent, serializable to dict
Error recording — exceptions captured in metrics, not just raised
Token bucket algorithm — sliding window, not fixed window
Key Design Decisions
Why simulate word-by-word streaming instead of true token streaming?
HuggingFace transformers doesn't support true token streaming without the TextIteratorStreamer class and threading. For this demo the goal is showing the streaming infrastructure — rate limiting, cancellation, TTFT measurement — not the model internals. The word-by-word simulation produces identical UI behavior. The code comment is explicit: "simulated — realistic for demo". In production you'd swap in TextIteratorStreamer.
Why a token bucket rate limiter instead of a simple counter?
A simple counter resets at fixed intervals — a user could send 10 requests at 11:59:59 and 10 more at 12:00:01, effectively getting 20 in 2 seconds. A token bucket (sliding window) tracks individual request timestamps and checks how many occurred in the last N seconds — much fairer and harder to abuse. The thread lock is critical: without it, concurrent requests could both pass the check before either increments the counter.
Why store only the last 50 requests in MetricsStore?
An unbounded history would grow forever in a long-running process, consuming memory. 50 requests is enough to compute meaningful p95 statistics (needs at least 20 data points) and show recent trends without memory pressure. The pop(0) approach is simple but O(n) — in production you'd use a collections.deque(maxlen=50) for O(1) append and eviction.
