What is MCP?

MCP (Model Context Protocol) is an open standard by Anthropic for connecting LLMs to external tools and data sources. Instead of hardcoding tool calls into your LLM app, MCP lets you build a standalone server that any MCP-compatible client (Claude Desktop, Claude Code, custom apps) can connect to and use.

The key insight: the LLM decides which tools to call based on the user's question. Ask "What's burning right now?" and Claude calls query_alerts then get_slo_status automatically — you don't write that logic, the model figures it out.

Architecture
How it works
  • 1
    Client connects — Claude Desktop spawns the MCP server as a subprocess via stdio. The server reads JSON-RPC messages from stdin and writes responses to stdout.
  • 2
    tools/list — client sends a tools/list request. Server responds with all 6 tool names, descriptions, and JSON Schema for arguments. Claude now knows what tools exist.
  • 3
    User asks a question — "What's burning right now?" Claude decides to call query_alerts then get_slo_status for each critical service — automatically, no code needed.
  • 4
    tools/call dispatched — server routes to the correct Python function by tool name. Arguments validated against JSON Schema before execution.
  • 5
    @timed_tool decorator — wraps every tool function. Records start time, executes the function, measures latency, calls audit() with result. Tool author writes zero logging code.
  • 6
    JSON-RPC response — tool result returned as JSON-RPC 2.0 result object. Claude receives the data, synthesizes a natural language response to the user.
6 SRE Tools

Each tool is registered with a name, description, and JSON Schema for argument validation. The MCP server advertises them via tools/list so any client knows what's available:

get_slo_status
SLO burn rate and error budget for a service. Returns status CRITICAL/WARNING/ELEVATED/OK with burn rate, budget remaining %, and recommended action.
service_name: str
get_runbook
Fetch runbook steps for an alert. Fuzzy match on alert name — "HighErrorRate" or "high error rate" both work. Returns immediate steps and escalation path.
alert_name: str
query_alerts
Recent firing alerts filtered by severity. Returns total count and structured alert list with service, duration, and summary.
severity: all|critical|warning
create_incident
Create a new incident record in the in-memory store. Generates INC-XXXXXX ID, sets status to "investigating", initializes timeline.
title, severity, service, description
get_incident_timeline
Full timeline of events for an incident — alert fired, on-call paged, acknowledged, root cause found, resolution. Falls back to mock timeline for demo IDs.
incident_id: str
summarize_incident
LLM-powered — Qwen2.5-0.5B generates a 3-sentence incident summary from the timeline. Covers what happened, root cause, and resolution. Heuristic fallback in CI.
incident_id: str
JSON-RPC 2.0 Protocol

MCP uses JSON-RPC 2.0 over stdio. Here's a real tool call exchange: 

→ Client sends tools/call request
{
  "jsonrpc": "2.0",
  "id": 3,
  "method": "tools/call",
  "params": {
    "name": "get_slo_status",
    "arguments": { "service_name": "auth-service" }
  }
}

← Server responds with tool result
{
  "jsonrpc": "2.0",
  "id": 3,
  "result": {
    "content": [{
      "type": "text",
      "text": "{\"service\": \"auth-service\", \"status\": \"WARNING\",
               \"burn_rate\": 4.61, \"budget_remaining_pct\": 90.8,
               \"action\": \"Investigate soon\"}"
    }]
  }
}

← Claude synthesizes: "auth-service is WARNING with 4.6x burn rate.
   You have 90.8% error budget remaining but should investigate soon."
Audit Logging

Every tool call is logged automatically via the @timed_tool decorator — zero effort for tool authors. The audit log is postmortem-ready: 

AUDIT_LOG — last 200 entries, in-memory

2026-06-10T03:12:44Z  query_alerts       ✓ success  2ms   {severity: "critical"}
2026-06-10T03:12:45Z  get_slo_status     ✓ success  1ms   {service_name: "auth-service"}
2026-06-10T03:12:46Z  get_runbook        ✓ success  1ms   {alert_name: "HighErrorRate"}
2026-06-10T03:12:51Z  create_incident    ✓ success  3ms   {title: "auth-service high error rate", severity: "critical"}
2026-06-10T03:13:02Z  summarize_incident ✓ success  33241ms  {incident_id: "INC-A3F2C1"}

The audit trail answers the postmortem question: "exactly what did the AI look at and in what order?" — something most AI tools can't answer. Each entry has a UUID, ISO timestamp, tool name, arguments, success flag, and latency.

Claude Desktop Integration

Add this to claude_desktop_config.json and Claude Desktop can call your SRE tools in any conversation: 

// ~/Library/Application Support/Claude/claude_desktop_config.json
{
  "mcpServers": {
    "sre-ops": {
      "command": "python",
      "args": ["-m", "src.mcp_server"],
      "cwd": "/path/to/p12-sre-mcp"
    }
  }
}

// Now ask Claude anything:
// "What services are burning right now?"
// "Show me the runbook for HighErrorRate"
// "Create an incident for auth-service"
// "Summarize incident INC-A3F2C1"

Key Design Decisions
Why MCP instead of a custom tool-use API?
MCP is an open standard — any MCP client works without writing integration code. Claude Desktop, Claude Code, and custom apps built on Claude all support MCP natively. A custom API would require writing a new integration for every client. MCP gives you the integration for free once the server is built.
Why @timed_tool decorator instead of logging inside each function?
A decorator separates the audit concern from the business logic. Each tool function does exactly one thing — answer the SRE question. The decorator handles timing and logging transparently. Adding a new tool automatically gets audit logging with zero extra code. This is the same pattern as middleware in web frameworks.
Why JSON-RPC 2.0 instead of REST?
MCP uses JSON-RPC 2.0 because it maps naturally to the request/response pattern over stdio — each message has an id, method, and params. REST assumes HTTP verbs and URLs which don't translate to stdin/stdout. JSON-RPC also supports notifications (server-to-client messages without a request) which MCP uses for progress updates.
Why include summarize_incident as an MCP tool instead of letting Claude summarize?
Exposing summarize_incident as a tool means the summary is generated server-side with the full timeline context, logged in the audit trail, and returnable as structured data. If Claude summarizes directly it's not logged, not reproducible, and not auditable. For incident management, every action taken — including AI summaries — should be in the audit log.
